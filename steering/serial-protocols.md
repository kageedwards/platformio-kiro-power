# Serial Communication Protocols for Embedded Development

Best practices and standards for hardware serial communication protocols. Load this when working with buses, peripherals, or inter-device communication.

## UART / USART

Universal Asynchronous (or Synchronous/Asynchronous) Receiver-Transmitter. The most common serial interface on microcontrollers.

### Configuration Parameters
- **Baud rate:** Must match on both ends. Common: 9600, 19200, 38400, 57600, 115200, 230400, 460800, 921600
- **Data bits:** 5, 6, 7, or 8 (8 is standard)
- **Parity:** None, Even, Odd (None is most common)
- **Stop bits:** 1 or 2 (1 is standard)
- **Flow control:** None, RTS/CTS (hardware), XON/XOFF (software)

### Best Practices
- Always configure both ends identically. Mismatched baud rates produce garbled data with no error indication.
- Use hardware flow control (RTS/CTS) for high-speed or high-volume transfers to prevent buffer overruns.
- On AVR/Arduino, `Serial.begin(baud)` configures UART. On ESP32, you have three hardware UARTs (UART0, UART1, UART2) with configurable pins.
- Buffer sizes matter on constrained MCUs. AVR default is 64 bytes. Increase with `-D SERIAL_RX_BUFFER_SIZE=256` build flag if needed.
- Always check `Serial.available()` before `Serial.read()`. Never assume data is present.
- Use timeouts (`Serial.setTimeout()`) to prevent blocking reads from hanging your main loop.
- For binary protocols, use `Serial.readBytes()` with explicit length rather than `Serial.readString()`.
- Flush transmit buffer with `Serial.flush()` before sleep or power-down modes.

### Platform-Specific Notes
- **AVR (ATmega):** Single or dual hardware UART. Software serial available but CPU-intensive and unreliable above 57600 baud.
- **ESP32:** Three hardware UARTs. UART0 is typically used for USB/programming. Pins are remappable via `HardwareSerial::begin(baud, config, rxPin, txPin)`.
- **STM32:** Multiple USARTs with DMA support. Use DMA for high-throughput applications.
- **RISC-V (SiFive):** UART peripheral varies by SoC. Check the specific SoC datasheet for register layout.
- **PIC:** EUSART module. Configuration via TXSTA, RCSTA, and SPBRG registers.

## RS-232

Electrical standard for UART signaling at higher voltages (±3V to ±15V). Used for longer-distance communication and legacy equipment.

### Best Practices
- RS-232 uses inverted logic: logic 1 = -3V to -15V, logic 0 = +3V to +15V. MCU UART is TTL (0V/3.3V or 0V/5V).
- **Always use a level shifter** (MAX232, MAX3232) between MCU UART and RS-232 devices. Direct connection will damage the MCU.
- MAX3232 for 3.3V MCUs, MAX232 for 5V MCUs. Both need external capacitors (check datasheet for values).
- Cable length: up to ~15 meters at 19200 baud. Reduce baud rate for longer runs.
- For noise immunity on long cables, use shielded cable and connect shield to ground at one end only.
- DB-9 pinout: Pin 2 = RX, Pin 3 = TX, Pin 5 = GND (DTE perspective). Null modem cables swap TX/RX.
- Always verify TX/RX crossover. DTE-to-DTE requires null modem; DTE-to-DCE uses straight-through.

## SPI (Serial Peripheral Interface)

Synchronous, full-duplex, controller-peripheral bus. Fast (up to 80+ MHz on some MCUs), but requires more pins.

### Signal Lines

Per the [OSHWA resolution to redefine SPI signal names](https://www.oshwa.org/a-resolution-to-redefine-spi-signal-names/), prefer the updated terminology:

- **SDO** (Serial Data Out) — data from controller to peripheral. Legacy name: MOSI.
- **SDI** (Serial Data In) — data from peripheral to controller. Legacy name: MISO.
- **SCK/SCLK** (Serial Clock) — generated by the controller
- **CS** (Chip Select) — active low, one per peripheral device

> **Note:** Many existing libraries, datasheets, and schematics still use the legacy MOSI/MISO/SS naming. When interfacing with these, map the signals correctly: SDO↔MOSI, SDI↔MISO, CS↔SS. The electrical function is identical — only the naming differs.

### Configuration Parameters
- **Clock speed:** Device-specific maximum. Start slow (1 MHz), increase after verifying.
- **SPI Mode** (Clock Polarity + Phase):
  | Mode | CPOL | CPHA | Description |
  |------|------|------|-------------|
  | 0 | 0 | 0 | Clock idle low, sample on rising edge |
  | 1 | 0 | 1 | Clock idle low, sample on falling edge |
  | 2 | 1 | 0 | Clock idle high, sample on falling edge |
  | 3 | 1 | 1 | Clock idle high, sample on rising edge |
- **Bit order:** MSB first (most common) or LSB first

### Best Practices
- **Always check the peripheral's datasheet** for supported SPI mode and max clock speed. Using the wrong mode produces garbage data.
- Keep SPI traces short. Signal integrity degrades with length, especially at high clock speeds.
- Use dedicated CS pins per peripheral. Never share CS lines.
- Pull CS high (inactive) for all peripherals during initialization before configuring SPI.
- On Arduino/PlatformIO, use `SPI.beginTransaction(SPISettings(speed, bitOrder, mode))` and `SPI.endTransaction()` to safely share the bus between libraries.
- DMA-based SPI transfers are strongly recommended on STM32 and ESP32 for large data blocks (displays, SD cards, flash memory).
- When using multiple SPI devices, ensure only one CS is active at a time. Forgetting to deassert CS is the most common SPI bug.
- For ESP32: VSPI (default) and HSPI are available. VSPI pins: SDO(23), SDI(19), SCK(18), CS(5).
- For AVR: SPI pins are fixed (SDO=11, SDI=12, SCK=13 on Uno/Mega). CS pin (10 on Uno, 53 on Mega) must be OUTPUT even if not used as chip select.

### Common SPI Peripherals
- SD cards (SPI mode 0, up to 25 MHz in SPI mode)
- Display controllers (ILI9341, SSD1306, ST7789)
- LoRa radios (SX1276, SX1278, SX1262)
- Flash memory (W25Q series)
- ADCs/DACs (MCP3008, MCP4922)

## I2C / TWI (Two-Wire Interface)

Synchronous, half-duplex, multi-controller bus. Uses only two wires but is slower than SPI.

### Signal Lines
- **SDA** (Serial Data) — bidirectional
- **SCL** (Serial Clock) — driven by the controller

### Configuration Parameters
- **Address:** 7-bit (standard) or 10-bit. Each device has a fixed or configurable address.
- **Clock speed:**
  - Standard mode: 100 kHz
  - Fast mode: 400 kHz
  - Fast mode plus: 1 MHz
  - High speed: 3.4 MHz (rarely used in hobby/maker contexts)

### Best Practices
- **Pull-up resistors are mandatory** on both SDA and SCL. Typical values: 4.7kΩ for 100 kHz, 2.2kΩ for 400 kHz. Too high = slow rise times; too low = excessive current.
- Only one set of pull-ups per bus. Many breakout boards include pull-ups — check for conflicts when connecting multiple devices. Disable extras by cutting traces or desoldering resistors.
- Use `Wire.begin()` for controller mode (no address), `Wire.begin(address)` for target/peripheral mode.
- Always check return values from `Wire.endTransmission()`:
  - 0 = success
  - 1 = data too long for buffer
  - 2 = NACK on address (device not found)
  - 3 = NACK on data
  - 4 = other error
- Scan for devices before assuming addresses:
  ```cpp
  for (uint8_t addr = 1; addr < 127; addr++) {
      Wire.beginTransmission(addr);
      if (Wire.endTransmission() == 0) {
          Serial.printf("Device found at 0x%02X\n", addr);
      }
  }
  ```
- Bus lockup recovery: If SDA is stuck low, toggle SCL manually (9 clock pulses) to release the bus. Some MCUs have built-in bus recovery.
- Keep bus length under 1 meter for reliable operation. Use I2C bus extenders (P82B715) for longer runs.
- On ESP32, I2C pins are configurable: `Wire.begin(SDA_PIN, SCL_PIN)`.
- On AVR, I2C pins are fixed: SDA=A4, SCL=A5 (Uno); SDA=20, SCL=21 (Mega).

### Address Conflicts
Common I2C addresses to watch for conflicts:
- 0x27 / 0x3F — PCF8574 LCD backpack
- 0x3C / 0x3D — SSD1306 OLED
- 0x48-0x4F — ADS1115 ADC
- 0x68 — MPU6050 IMU, DS3231 RTC
- 0x76 / 0x77 — BME280/BMP280

Use I2C multiplexers (TCA9548A) when address conflicts are unavoidable.

## USB (as Serial Communication)

USB CDC (Communications Device Class) provides virtual serial port over USB.

### Best Practices
- USB CDC is not true UART. Baud rate settings in `Serial.begin()` are often ignored on native USB MCUs (e.g., ATmega32U4, SAMD21, ESP32-S2/S3, RP2040).
- On boards with native USB (Leonardo, Micro, ESP32-S3), use `while (!Serial) { delay(10); }` after `Serial.begin()` to wait for the host connection.
- USB enumeration takes time. Add a startup delay (1-2 seconds) before sending data if the host needs to open the port.
- For ESP32-S2/S3 with native USB: `Serial` is USB CDC, `Serial0` is hardware UART0.
- DTR/RTS signals are used by many boards for auto-reset during upload. If serial monitor interferes with upload, configure `monitor_dtr = 0` and `monitor_rts = 0` in `platformio.ini`.
- USB HID (Human Interface Device) is an alternative for sending structured data without a serial driver. Libraries like TinyUSB support this.

## 1-Wire

Single-wire bidirectional bus (plus ground). Used primarily by Dallas/Maxim sensors.

### Best Practices
- Requires a 4.7kΩ pull-up resistor on the data line.
- Each device has a unique 64-bit ROM code. Use `OneWire` library's search function to enumerate devices.
- Parasitic power mode (power over data line) is unreliable with long cables or many devices. Use dedicated VCC when possible.
- Timing-critical protocol. Interrupts can cause read failures. On AVR, the OneWire library disables interrupts during bit operations.
- Maximum recommended bus length: ~100 meters with proper pull-up and low device count.
- Common devices: DS18B20 (temperature), DS2413 (GPIO), iButton (authentication).

## General Cross-Protocol Best Practices

1. **Always read the peripheral datasheet** before writing driver code. Don't rely on example code alone — it may be for a different revision or configuration.
2. **Start with the slowest supported speed** and increase only after verifying correct operation.
3. **Use logic analyzers** (Saleae, DSLogic, or even cheap $10 analyzers) to debug protocol issues. They're invaluable for verifying timing, bit order, and signal integrity.
4. **Decouple power** to peripherals with 100nF ceramic capacitors close to VCC/GND pins. Add 10µF bulk capacitors for devices with high transient current draw.
5. **Level shifting** is required when connecting 3.3V and 5V devices. Use bidirectional level shifters (BSS138-based modules) for I2C, and unidirectional for SPI/UART.
6. **Ground connections** must be solid and shared between all communicating devices. Floating grounds cause mysterious intermittent failures.
7. **Interrupt-driven or DMA-based** communication is preferred over polling for production firmware. Polling wastes CPU cycles and can miss data.
